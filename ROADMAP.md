# SleepingKnights 開発ロードマップ

**目的:** Go でインタプリタを作り、REPL とスクリプト実行を実現する。その後 LLVM ベースのネイティブコンパイラへ発展させ、最終的にコンパイラのセルフホスティングを達成する。

**注:** 本ドキュメントでは、開発中言語のコードネームを **SleepingKnights** として統一する。

## 開発条件

| 項目 | 内容     |
| ---- | -------- |
| 体制 | 個人開発 |

## 言語方針

| 項目       | 内容                                                 |
| ---------- | ---------------------------------------------------- |
| パラダイム | 手続き型 → OOP（ダックタイピング）→ 関数型へ段階拡張 |
| 型システム | 型推論付き静的型付け                                 |
| 初期対応OS | macOS                                                |

## MVP優先機能

- 必須: 手続き型の最小構文（変数・関数・if/loop）
- 実行方式: REPL + スクリプトファイル

## 最終ゴール

自作言語で書いたコンパイラが、自分自身をビルドできる状態を達成する。

## フェーズ別ロードマップ

### Phase 0: 最小仕様の確定（手続き型のみ）

- 成果物: 言語仕様 v0.1
- 手続き型 MVP の最小セットを定義（式・文・関数・if/loop）。
- 段階拡張前提で仕様を分割（Core / OOP / Functional）。
- 型推論ルール（ローカル推論中心）と型エラー表示方針を定義。
- AST 設計と演算子優先順位を確定。

### Phase 1: Go 製インタプリタ基盤（手続き型コア）

- 成果物: lexer/parser/evaluator
- Lexer/Parser 実装と AST 生成。
- 環境（スコープ）管理、関数呼び出し、制御構文評価。
- 単体テストで手続き型コア（式・分岐・ループ・関数）を固定化。

### Phase 2: 実行 UX（REPL / スクリプト実行）

- 成果物: CLI 実行系
- `slk` を引数なしで実行すると対話モード（REPL）を起動。
- `slk file.sk` のようにスクリプトファイルを渡すと実行。
- 構文エラー位置（行・列）を表示。

### Phase 3: 型検査・推論の導入（Core 安定化）

- 成果物: typer
- 型変数・単一化（unification）による型推論を実装。
- 関数シグネチャ推論と基本型（int/bool/string）をサポート。
- 型エラーを実行前に検出し、可読性の高いメッセージを返す。

### Phase 4: OOP 拡張（ダックタイピング導入）

- 成果物: OOP サブセット
- 構造的インターフェース的なダックタイピング規則を導入。
- オブジェクト生成、メソッド呼び出し、委譲の最小機能を実装。
- 手続き型コアとの互換性を保つ回帰テストを追加。

### Phase 5: 関数型拡張（段階導入）

- 成果物: FP サブセット
- 高階関数、クロージャ、イミュータブル志向のデータ操作を導入。
- 必要最小限の標準機能（文字列/コレクション/入出力）を整備。
- 手続き/OOP/FP 混在コードでの意味論を回帰テストで保証。

### Phase 6: LLVM コンパイラ 1st

- 成果物: ネイティブ実行可能ファイル
- AST/IR から LLVM IR 生成（手続き型コア + OOP/FP の必要サブセット）。
- Go 実装から実行ファイルを生成し、macOS で動作確認。
- インタプリタとの意味論差分を潰す回帰テストを作成。

### Phase 7: ブートストラップ開始

- 成果物: コンパイラの一部を自作言語化
- コンパイラのフロントエンド（例: parser/typer）から順に移植。
- Go 版を信頼コンパイラとして比較実行し、生成結果一致を確認。

### Phase 8: セルフホスティング達成

- 成果物: self-hosting build
- 自作言語版コンパイラで、自作言語版コンパイラを再ビルド。
- ハッシュ比較やスモークテストで再現性を確認。
- 達成条件: 自己再ビルド成功 + サンプルプログラム群の全通過。

## 完了判定チェックリスト

- REPL とスクリプト実行が安定動作する。
- 手続き型コアから OOP/FP 拡張まで後方互換を維持している。
- 型エラーを実行前に検出できる。
- LLVM 経由でネイティブ実行可能ファイルを出力できる。
- 自作言語版コンパイラで自己再ビルドが通る。
